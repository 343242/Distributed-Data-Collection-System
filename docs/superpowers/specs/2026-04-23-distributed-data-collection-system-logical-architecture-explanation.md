## 1. 文档目的与阅读方式

这份文档不是新的架构方案，也不是对原逻辑架构方案的替代版本。它的目标只有一个：

**把现有逻辑架构方案背后的设计理由解释清楚。**

它主要面向以下读者：

- 后续参与实现的开发人员
- 负责联调和排障的工程人员
- 后续接手系统维护和迭代的维护人员

阅读这份文档时要带着两个问题：

1. 这套系统是怎么运行起来的？
2. 为什么架构方案要这样分层、这样流转、这样约束？

因此，这份文档采用“**按系统主线展开 + 每一节落架构决策说明**”的写法。  
也就是说，每一节都同时回答三件事：

- 这一层在系统里是怎么工作的
- 为什么方案选择这样设计
- 如果不这么做，会出现什么问题

## 2. 系统到底在解决什么问题

这个系统表面上看像“多个采集程序把数据写进一个中心数据库”，但实际上它要解决的是一个更复杂的问题组合：

- 数据源分散在客户现场
- 同一个客户现场可能有多个 `Agent`
- 不同数据源需要不同类型的 `Connector`
- 中心侧必须统一下发配置、统一监控、统一控制
- 网络不是完全稳定的
- 数据允许重复，但尽量不能丢失

如果把这个系统设计成“几个采集脚本 + 一个中心数据库”，短期内也许能跑，但很快会遇到以下问题：

- 无法统一管理不同类型 Connector 的程序包和版本
- 一旦网络波动，数据容易丢失或重复失控
- 现场节点和中心逻辑耦合太深，排障困难
- 后续扩容时无法判断到底是控制链路有问题，还是数据链路有问题

所以，逻辑架构方案从一开始就不是围绕“怎么快速采到数据”来设计，而是围绕下面三个目标来设计：

- **可管理**：中心能够统一控制现场运行状态
- **可恢复**：节点异常、网络中断、下游变慢时系统还能继续工作
- **可演进**：后续新增 Connector 类型、扩展 Agent 数量、增强迁移能力时，不需要推倒重来

### 如果不这么做，会出现什么问题

- 系统最初可能看起来简单，但一进入真实运行环境就会因为现场分散、网络不稳、数据源多样而迅速失控
- 开发阶段省掉的边界和约束，后续会在排障、补偿、重试和运维中成倍补回来

## 3. 为什么系统要分成这几个角色

逻辑架构里最终固定了 5 个核心角色：

- `Business System`
- `Data Gateway`
- `Client Agent`
- `Connector`
- `Data Storage`

这个划分不是为了“画图好看”，而是为了把不同问题交给不同角色处理。

### 3.1 Business System

`Business System` 只负责：

- 配置任务
- 查看状态
- 查询结果

它不负责现场执行，不负责直接采数据，也不负责做可靠传输。

这样设计的原因是：业务系统的职责是“表达业务意图”，不是“承担采集执行”。  
如果让业务系统直接参与执行层细节，后续会把任务定义、运行态管理、异常恢复全部混在一起。

### 3.2 Data Gateway

`Data Gateway` 是中心控制和中心接入枢纽。  
它既要向下管理所有 Agent，又要向上向业务系统提供统一控制入口，同时还要承接上传数据。

之所以把 Gateway 放在中心，而不是让 Agent 之间直接协同，是因为：

- 中心必须掌握全局任务和节点状态
- 程序包、配置、审计和监控都需要统一管理
- 数据接入需要统一边界，不能让每个 Agent 自己定义“什么时候算成功”

### 3.3 Client Agent

`Client Agent` 是客户现场的宿主和控制器，不是简单的消息转发器。  
它负责：

- 在本机托管多个 Connector 实例
- 管理程序包和实例生命周期
- 本地可靠落盘
- 上传数据
- 监控和恢复

这是一个非常关键的设计决定，因为现场侧真正面对的是“节点本地故障 + 网络不稳定 + 多实例共存”这些问题。  
这些问题只能由现场宿主来解决，不能指望中心替代。

### 3.4 Connector

`Connector` 是采集执行单元，只负责连接源端、读取数据、解析协议、生成统一采集消息。  
它不负责：

- 全局调度
- 跨网络可靠传输
- 最终入库
- 中心控制逻辑

这样做的目的是把 Connector 的复杂度压在“源端采集”这一件事情上，避免它变成“采集 + 管理 + 传输 + 恢复”四合一程序。

### 3.5 Data Storage

`Data Storage` 被拆成两个逻辑层次：

- `Durable Ingest Storage`
- `Business Query Storage`

这个分层后面会专门展开。这里先给结论：  
**中心第一可靠落点** 和 **最终查询存储** 不是一回事，不能用一个“数据库”概念糊过去。

### 如果不这么做，会出现什么问题

- 业务系统会承担执行职责，导致配置、执行、恢复混在一起
- Agent 会退化成薄转发层，现场可靠性没人兜底
- Connector 会承担过多责任，导致类型越多越难维护
- 数据接入成功、处理成功、入库成功三个边界会混成一件事

## 4. 为什么 Gateway 要拆成 Control Plane 和 Data Plane

这是整个架构方案中最重要的一个拆分。

### 4.1 它们分别负责什么

`Control Plane` 负责：

- 任务管理
- Agent 管理
- 程序包与配置管理
- 命令编排
- 监控与审计

`Data Plane` 负责：

- 接收上传数据
- 接入确认
- 持久化接入
- 标准化
- 去重
- 路由
- 最终入库

### 4.2 为什么必须拆

控制面和数据面的工作特点完全不同：

- 控制面强调一致性、版本控制、状态管理、命令串行化
- 数据面强调吞吐、缓冲、异步处理、失败隔离和背压

如果不拆，后果通常是：

- 数据高峰把控制面拖慢
- 控制逻辑和数据处理互相污染
- 很难定义清楚“命令冲突”和“数据积压”到底属于哪一层问题

### 4.3 为什么第一版定成单 Leader

第一版明确采用：

- `Control Plane` 写路径：`单 Leader + 共享元数据存储`

原因很直接：

- 控制对象是任务、配置、实例、命令，这些都要求串行一致的状态推进
- 第一版最需要避免的是“多个控制面实例同时改同一对象”
- 比起一开始做复杂分布式协调，单 Leader 更稳、更容易验证

这并不意味着系统不能高可用，而是说第一版先把**控制写冲突**的问题压住，再谈横向扩展。

### 如果不这么做，会出现什么问题

- 控制命令可能冲突
- 数据高峰时连心跳、同步、配置下发都会一起变慢
- 排障时无法区分问题是出在控制面还是数据面

## 5. 为什么 Agent 不是简单执行器，而是现场宿主

`Agent` 在这套架构里承担的是“现场控制器”角色，而不是单纯的连接节点。

### 5.1 它要解决什么问题

现场侧有几个中心侧替代不了的问题：

- Connector 程序包需要在本机下载、校验、安装
- 同一台机器上可能运行多个实例
- 网络断开时数据必须先保住
- 节点重启后需要恢复运行状态和未完成消息

这些事情不适合放到 Connector 里，更不适合让 Gateway 远程代劳。

### 5.2 为什么本地可靠性必须落在 Agent

方案要求：

- Connector 采到的数据先写入 Agent 本地持久化队列
- `Checkpoint` 和消息入队在 Agent 本地原子提交

这样做的核心逻辑是：

- Connector 负责知道“采到了哪里”
- Agent 负责保证“采到的数据真的先保住了”

这条边界非常重要。因为真正危险的不是“重复采到”，而是“还没保住数据，进度却已经往前走了”。

### 5.3 为什么要独立进程托管 Connector

方案规定每个 `Connector Instance` 默认独立进程运行。  
这样做是为了：

- 故障隔离
- 资源限制
- 日志隔离
- 单实例重启

如果所有 Connector 都在 Agent 内部线程里跑，一个实例的崩溃、阻塞、内存泄漏都可能拖垮整个 Agent。

### 如果不这么做，会出现什么问题

- 现场节点会变成无状态薄节点，网络一抖就丢数
- Connector 既要采集又要管恢复，复杂度快速失控
- 一个实例异常影响整机所有采集任务

## 6. 为什么 Connector 只负责采，不负责全局管理

这个边界看起来简单，但实际上非常关键。

### 6.1 Connector 的职责为什么要收敛

`Connector` 负责的是“源端语义”：

- 怎么连源端
- 怎么鉴权
- 怎么读取
- 怎么解析
- 怎么表达 checkpoint

它最了解源系统，所以它负责定义“位点语义”和“记录去重语义”。

但它不应该负责：

- 全局任务调度
- Agent 级可靠队列
- 中心侧接入确认
- 最终存储结构

### 6.2 为什么这样拆更稳

因为源端采集本来就已经足够复杂：

- 数据库类源有增量位点问题
- 文件类源有轮转问题
- CDC 类源有游标和快照问题
- API 类源有 cursor、分页和限流问题

如果再把中心协调和可靠传输加进去，Connector 很快会变得难以复用，也很难稳定。

### 如果不这么做，会出现什么问题

- 不同类型 Connector 会各自实现一套上传、恢复、幂等、重试逻辑
- 整个系统的可靠性语义会被分散在多个程序包里，最后无法统一治理

## 7. 控制链路为什么这样设计

控制链路是：

`Business System -> Gateway Control Plane -> Agent -> Connector`

这里有两个关键设计决定。

### 7.1 为什么采用目标状态优先

方案强调：

- `desiredState` 是事实来源
- `command` 是收敛信号

原因是：  
如果只靠命令驱动，一旦 Agent 离线、重连、延迟或者消息乱序，就很容易出现：

- 老命令覆盖新配置
- 过期命令重新执行
- 系统不知道当前“应该是什么状态”

而目标状态模型的优势在于：

- 先定义系统应该收敛到什么状态
- 命令只是加快收敛
- 重连时只需要同步最新目标状态，不需要回放一堆历史命令

### 7.2 为什么控制面采用 Agent 主动拉

方案采用：

- 控制面：Agent 主动拉
- 数据面：Agent 主动推

控制面之所以让 Agent 主动拉，是因为现场节点通常位于客户网络侧，中心不适合反向主动打进去。  
主动拉的好处是：

- 网络模型简单
- 重连恢复容易
- 不依赖 Gateway 主动连现场
- 更适合目标状态同步

### 如果不这么做，会出现什么问题

- 控制命令会变成时效不清、乱序难控的消息流
- Agent 重连后很难判断该执行哪些历史命令
- 现场网络边界会让控制通道变得不稳定

## 8. 数据链路为什么这样设计

数据链路是：

`Source -> Connector -> Agent Local Persistent Queue -> Gateway Ingress Persistent Queue / Durable Ingest Storage -> Processing -> Business Query Storage`

这个链路里最重要的是**确认边界**。

### 8.1 为什么先本地落盘

现场第一可靠落点是：

- `Agent Local Persistent Queue`

也就是说，Connector 采到数据之后，第一件事不是“赶紧上传”，而是“先让 Agent 保住”。  
因为网络问题是常态，上传失败是可恢复的，而本地还没保住就推进进度是不可接受的。

### 8.2 为什么 Gateway 的 Accepted 不能等于写最终库成功

方案明确：

- Gateway 只有在数据进入中心第一可靠落点后才返回 `Accepted`
- `Accepted` 不等于已经写入最终查询库

这是一个典型的架构分层决策。  
如果把接收成功和最终入库绑在一起，那么：

- 最终库一慢，前面全链路一起堵
- Agent 会因为最终库抖动而不断重发
- 系统会把“接住数据”和“处理完数据”混成一个步骤

因此必须拆开：

- 先接住
- 再处理
- 再入库

### 如果不这么做，会出现什么问题

- 中心侧无法定义清楚“到底是不是已经接住了”
- 最终库一出问题，Agent 就会无休止重发
- 整条数据链路的恢复边界会变得非常模糊

## 9. 为什么可靠性要这样设计

系统的可靠性目标不是 `exactly-once`，而是：

- **至少一次**
- **允许重复**
- **尽量不丢**

这是一个很现实的取舍。

### 9.1 为什么允许重复

因为在分布式系统里，“完全不重复”往往意味着要付出非常高的复杂度和性能代价。  
对于这个系统来说，最重要的是：

- 现场不能因为网络抖动丢数据
- 中心不能因为下游慢导致已经接住的数据失效

所以方案选择：

- 传输层允许重复
- 业务层做幂等和去重

### 9.2 为什么需要两层 ID

系统定义了：

- `transportMessageId`
- `recordDedupKey`

两者解决的是不同问题：

- `transportMessageId` 解决“同一条上传消息是否被重复接收”
- `recordDedupKey` 解决“同一条源记录是否重复生效”

如果只用一个 ID，就会把“传输重复”和“业务重复”混在一起，后面很难处理。

### 9.3 为什么 Checkpoint 和消息入队必须原子提交

这是整个可靠性里最不能妥协的一条。

如果 checkpoint 先提交了，消息却没保住，系统就会：

- 进度已经往前
- 数据却永远丢失

因此方案明确要求：

- Agent 本地必须把“消息入队”和“checkpoint 更新”作为同一原子提交单元

这条约束保证了：

- 可以重复
- 但不能因为进度前推而丢数

### 如果不这么做，会出现什么问题

- 故障恢复时会出现“看起来进度正常，但数据其实没了”
- 这种问题最难排查，也最难补救

## 10. 为什么存储要分层

这是逻辑架构方案里后期才被明确收口的一条，但实际上非常关键。

### 10.1 为什么不能直接写最终业务库

如果系统直接把 Agent 上传的数据写到最终查询库，会有几个问题：

- 最终库写慢会反向影响数据接入
- 查询库和可靠落地点混成一个东西
- 难以做补偿、重放和失败隔离

所以方案最后定成两层：

- `Durable Ingest Storage`
- `Business Query Storage`

### 10.2 为什么第一版采用 Kafka + PostgreSQL

第一版明确采用：

- `Durable Ingest Storage`：`Kafka`
- `Business Query Storage`：`PostgreSQL`

原因是这两个组件分别很好地承担了不同职责：

`Kafka` 适合做：

- 中心第一可靠落点
- 顺序追加
- 保留与重放
- 将接入确认和后续处理解耦

`PostgreSQL` 适合做：

- 最终业务结果存储
- 幂等落库
- 事务能力
- 查询支撑
- 失败隔离表

这不是“技术流行度”的选择，而是职责匹配的选择。

### 10.3 为什么失败隔离单独保留

处理链失败后，不能简单丢弃消息。  
因为系统的目标是“允许重复，但尽量不丢”。所以：

- 可重试的重试
- 不可恢复的要进入失败隔离区

失败隔离的价值在于：

- 保留证据
- 支持补偿
- 支持人工排查
- 避免坏数据无限拖累主处理链

### 如果不这么做，会出现什么问题

- 一旦最终库抖动，中心接入就会不稳
- 失败消息没有归宿，要么丢弃，要么无限重试
- 后续补偿和审计没有依据

## 11. 为什么网络分区、离线策略和背压要这样处理

真实运行中，网络分区和下游变慢不是边缘场景，而是常见场景。

### 11.1 为什么离线策略不能一刀切

方案支持三种 `offlinePolicy`，并将默认值收口为：

- `ContinueUntilWatermark`

原因是不同任务对断网的容忍度不同：

- 有些任务断了以后可以补采
- 有些任务断了以后补不回来
- 有些任务可以继续采一阵，但不能把本地磁盘打满

所以统一“断链继续采”或统一“断链就停”都太粗。

默认采用 `ContinueUntilWatermark`，是因为它最平衡：

- 先保证数据连续性
- 再通过水位保护控制本地风险

### 11.2 为什么采用分级背压

方案采用的背压链路是：

`Storage Slowdown -> Gateway Processing Backlog -> Gateway Ingress Pressure -> Agent Upload Slowdown -> Agent Queue Growth -> Connector Throttle/Pause`

这背后的逻辑是：

- 每一层先吸收本层波动
- 超过阈值再向上游传递压力

这样能避免两个极端：

- 一出问题就全链路暂停
- 或者某一层无限堆积直到雪崩

### 如果不这么做，会出现什么问题

- Storage 一慢，所有采集立即抖动
- 或者 Gateway / Agent 本地队列无限增长，最后磁盘耗尽
- 背压行为不可预测，排障时难以判断压力从哪里开始传播

## 12. 为什么动态分发不纳入第一版核心交付

逻辑架构里保留了动态分发的完整语义，但明确：

- 第一版默认不实现动态分发执行能力

这是一个刻意的收敛，而不是能力缺失。

### 12.1 为什么要保留语义，但不先实现

动态分发看起来只是“把任务从一个 Agent 挪到另一个 Agent”，但实际上会牵涉：

- owner 切换
- drain
- checkpoint 交接
- 并发双采风险
- 迁移失败回退

这些都会直接碰到系统最脆弱的部分：数据完整性。

因此第一版选择：

- 先把 `Rebalancing`、单主归属、checkpoint 交接这些语义定下来
- 但不把实际迁移执行能力纳入核心交付

### 12.2 这样做的价值

这样做可以保证：

- 状态模型和接口先预留
- 后续增强时不需要推翻已有设计
- 第一版先把采集、接入、可靠性和存储这条主链做稳

### 如果不这么做，会出现什么问题

- 第一版会被过早拖入复杂迁移场景
- 迁移能力一旦没做稳，最容易直接造成重复失控或数据丢失

## 13. 第一版到底交付什么，不交付什么

这套逻辑架构方案最终收口的第一版核心交付是：

- 多 Agent 纳管
- Agent 宿主管理多个 Connector
- 程序包与配置管理
- 控制面目标状态同步
- 数据面接入确认
- Agent 本地可靠落盘
- 中心第一可靠落点
- 最终业务入库
- 幂等去重
- 失败隔离
- 基础监控与审计

第一版明确不纳入核心交付的内容是：

- 动态分发执行能力
- 双活迁移
- 同一采集流多 Agent 并行消费
- 无停机热迁移
- 精细化容量优化和分析型副本

这种边界收口的好处是：

- 开发团队知道什么必须先做稳
- 运维团队知道哪些能力不能默认存在
- 后续需求扩展时不会误以为系统已经支持更复杂的能力

### 如果不这么做，会出现什么问题

- 文档会看起来“什么都支持”，实际交付却和预期不一致
- 团队在实现阶段会不断把增强能力当成首版必做，导致项目失焦

## 14. 开发和维护人员最容易误解的点

### 14.1 `Accepted` 不等于已经写入最终业务库

它只表示：

- 数据已经进入中心第一可靠落点

### 14.2 `Connector` 不是自治节点

它必须运行在 Agent 的托管之下。  
不要把 Connector 设计成自己做全局管理、自己决定生命周期的独立节点。

### 14.3 `Gateway` 不是一个单体大服务的概念

逻辑上必须区分：

- `Control Plane`
- `Data Plane`

即使第一版部署时物理上离得近，也不能在逻辑设计里混成一团。

### 14.4 `Kafka` 不是最终业务存储

它解决的是“先接住、可重放”，不是“最终可查询”。

### 14.5 `Checkpoint` 不是简单 offset 数字

它是“采集进度 + 恢复上下文”的组合。  
尤其在文件、CDC、分页 API 场景里，这一点非常重要。

## 15. 总结：这套架构的核心取舍

这套架构方案的核心取舍可以概括成一句话：

**先把现场采集、中心接入和数据保住，再谈更复杂的弹性和高级能力。**

它优先保证的是：

- 角色清晰
- 边界清晰
- 状态可追踪
- 失败可恢复
- 数据尽量不丢

它刻意没有在第一版追求的是：

- 所有能力一步到位
- 复杂分布式迁移
- 端到端 exactly-once
- 控制与数据的高度自动弹性

这不是保守，而是有意的工程取舍。  
对于一个跨客户现场、跨网络边界、强调可靠采集的系统来说，**第一版最重要的不是“功能看起来多”，而是“核心链路足够稳”。**

只要理解了这一点，就能理解整份逻辑架构方案为什么会做出今天这些选择。
